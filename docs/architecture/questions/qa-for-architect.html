Как пользоваться: Это живой документ. Каждый вопрос имеет ответ и (при необходимости) анализ альтернатив. Если решение спорное — появится желтый блок «Альтернативы». Дополняйте по мере поступления новых вопросов.

1. Общая архитектура (Q1–Q6)

1Почему микросервисы, а не монолит?

Ответ: Микросервисы выбраны потому что у нас сложная партнёрская логика (3 уровня, ранги, бонусы), разные категории товаров и независимые команды (каталог vs заказы vs партнёрка). Каждый сервис масштабируется отдельно и не ломает остальные.

Альтернатива: Модульный монолит (NestJS с модулями) — дешевле и проще на старте. Если команда 1–2 человека на первые 3 месяца — монолит имеет смысл. Переход к микросервисам тогда будет рефакторингом.
2Где проходит граница между микросервисами?

Ответ: По доменам бизнеса: товары, заказы, партнёрка, выплаты, уведомления. Принцип: один сервис = одна бизнес-сущность + её данные.

3Зачем API Gateway вместо прямого доступа к сервисам?

Ответ: API Gateway (GraphQL) — единая точка входа. Клиент не знает, сколько сервисов за ним. Gateway агрегирует данные, кэширует, ограничивает запросы (rate limit), авторизует.

4Синхронные (HTTP) или асинхронные (очередь) вызовы между сервисами?

Ответ: На MVP — синхронные REST-вызовы. Проще в отладке, не нужна отдельная инфраструктура очередей.

Асинхронно (через Redis Streams / BullMQ) — только для:

  • Отправки уведомлений (email/SMS)
  • Пересчёта рангов (фоновая задача)
Альтернатива: RabbitMQ / Kafka сразу. Излишне для MVP — требует инфраструктуры, мониторинга, dead letter handling. Добавляем при реальных проблемах.
5Почему Headless CMS, а не классическая CMS?

Ответ: Классическая CMS (WordPress, Bitrix) привязана к фронтенду, сложнее кастомизировать партнёрский кабинет, тяжелее масштабировать. Headless CMS даёт только контент через API, фронтенд — полностью наш (Next.js).

6Что происходит, если микросервис упал?

Ответ: Несколько уровней защиты:

  • Circuit breaker — если сервис не отвечает N раз, прекращаем попытки и возвращаем fallback
  • Корзина в Redis — не зависит от сервиса заказов
  • Кэш каталога — даже если сервис товаров упал, пользователи видят закешированные товары
  • Graceful degradation — если не загрузились похожие товары, страница всё равно работает

2. Стек технологий (Q7–Q12)

7TypeScript на бэкенде и фронтенде — это преимущество или риск?

Ответ: Единый стек (Node.js + TS) — преимущество при небольшой команде: один язык, один линтер, переиспользование типов между фронта и бэкенда.

Риск: TypeScript не серебряная пуля. Нужно строго следить за strict mode, noImplicitAny, и использовать Zod для валидации входящих данных на runtime.

8Почему NestJS, а не Express или Fastify?

Ответ: NestJS предоставляет модульную архитектуру из коробки, валидацию через Pipes, Guards для авторизации, Interceptors для логирования. Express слишком низкоуровневый. Fastify быстрее, но NestJS работает на Fastify под капотом.

9Drizzle ORM или Prisma — почему Drizzle?

Ответ: Drizzle — легче (7kb vs 500kb у Prisma), быстрее (ближе к нативному SQL), типобезопасная. Скорость работы сервисов приоритетнее для мобильной аудитории.

Почему не Prisma: Prisma даёт более высокий уровень абстракции и проще для новичка. Но при работе с ИИ-ассистентом (который знает оба ORM) преимущества Prisma нивелируются — Drizzle проще понять, потому что ближе к SQL. Миграции в Drizzle — тоже есть (drizzle-kit).
10Next.js App Router или Pages Router?

Ответ: App Router (React Server Components) — серверный рендеринг более эффективный, меньше JS на клиенте, лучше для мобильных. Pages Router — legacy.

11Зачем Zustand, а не Redux?

Ответ: Zustand — 10 строк для store, нет провайдеров, работает с React Server Components. Для e-commerce на старте Redux — overkill.

12Почему мультиязычная архитектура (Node.js + Python)?

Ответ: Раньше был выбран единый стек (только Node.js) для простоты. Но при работе с ИИ-ассистентом (Qwen) аргумент «один язык для команды» отпадает — Qwen равно хорошо пишет на обоих.

Мультиязычная архитектура:

  • API Gateway, Product, Order, Payout, Notification — Node.js + TypeScript (I/O-heavy, SSR, GraphQL)
  • Partner Service — Python + FastAPI (графовые вычисления, обходы сети, пересчёт рангов)
  • Analytics Service — Python + FastAPI (pandas, статистика, будущие ML-модели)

Риски: Два стека зависимостей. Митигация: каждый сервис = отдельный Docker. docker-compose up поднимает всё одной командой.

3. Данные и база данных (Q13–Q17)

13PostgreSQL или MongoDB — почему реляционная?

Ответ: Партнёрская сеть — это граф связей с транзакционными гарантиями. Начисление бонусов, расчёт рангов, реферальные цепочки требуют ACID.

MongoDB отлично для каталога товаров, но не для финансовых транзакций.

14Отдельные БД или общая БД с разными схемами?

Ответ: На MVP — общая БД с отдельными схемами (product_service, order_service, partner_service). Это дешевле, проще бекапить, можно мигрировать к отдельным БД позже.

15Как работаем с миграциями БД?

Ответ: Drizzle migrate (drizzle-kit). Каждый сервис управляет своей схемой. Правило: миграция — только additive (add column, не drop). Breaking changes — в два этапа.

16Что произойдёт, если Redis упадёт?

Ответ: Корзина восстанавливается из БД. Сессии сбрасываются — пользователи перезайдут. Кэш товаров — сервис отвечает напрямую из PostgreSQL.

Внимание: Redis — single point of failure на MVP. На фазе 2 настраиваем Redis Sentinel или Redis Cluster.
17Где храним фото и видео товаров?

Ответ: Strapi Upload Provider -> S3-совместимое хранилище. На старте — локальное хранилище в Docker. На продакшене — Cloudflare R2 (дешевле AWS S3, нет egress-платежей).

4. API и коммуникация (Q18–Q21)

18GraphQL сразу или начать с REST и перейти потом?

Ответ: GraphQL сразу. Большинство пользователей — с мобильных устройств. Один запрос = все данные для страницы. Нет overfetching/underfetching. Переходить с REST на GraphQL на масштабе — дорого и рискованно.

REST остаётся для: внутренней коммуникации между сервисами, вебхуков от внешних систем.

19Версионирование API — как и когда?

Ответ: GraphQL не версионируется — добавляем поля, деприкейтим старые. Breaking changes — через маркировку @deprecated + период совместимости.

REST вебхуки — версируются в URL: /webhooks/yookassa/v1.

20Как сервисы внутри аутентифицируют друг друга?

Ответ: На MVP — внутренний JWT-токен, выдаваемый при старте сервиса. Сервисы в одном Docker Compose используют shared secret.

На фазе 2–3: mTLS между сервисами или API Gateway с OAuth2/OIDC.

21Единый формат ошибок или каждый сервис по-своему?

Ответ: Единый формат для GraphQL: { errors: [{ message, code, path }] }. HTTP-коды стандартные (400, 401, 403, 404, 422, 500). Internal error codes — уникальные с префиксом домена (PARTNER_001, ORDER_002).

5. Безопасность (Q22–Q25)

22Как защищены от SQL-инъекций и XSS?

Ответ:

  • Drizzle ORM — параметризированные запросы, SQL-инъекции невозможны на уровне ORM
  • React/Next.js автоматически экранирует XSS
  • Zod для валидации всех входящих данных
23JWT или сессии для аутентификации?

Ответ: Гибрид. Access token (JWT, 15 мин) — в памяти клиента. Refresh token (httpOnly cookie, 30 дней) — в Redis с возможностью отзыва.

Альтернатива: Чистые сессии в Redis — проще отзывать, но привязывают к состоянию. JWT stateless — лучше для масштабирования.
24Как соблюдаем PCI DSS?

Ответ: Не храним платёжные данные вообще. ЮKassa обрабатывает карты, мы получаем только статус оплаты. Это снимает 90% PCI DSS-требований.

25Как защищены от фрода в партнёрской сети?

Ответ: Hold-период бонусов (7–14 дней), проверка по IP/устройству (запрет саморефералов), лимит на возвраты (flag для ручной проверки), минимальная сумма вывода бонусов.

6. Производительность (Q26–Q28)

26Какой целевой TTFB (Time to First Byte)?

Ответ: Для SSR-страниц (каталог, товар): менее 200ms. Для API: менее 100ms (p95). Для GraphQL-запросов: менее 150ms.

27Стратегия кэширования?

Ответ: Многоуровневое кэширование:

  • Redis — кэш ответов API товаров (TTL 5–15 минут)
  • Next.js ISR — Incremental Static Regeneration (регенерация страницы раз в N минут)
  • CDN (Vercel) — кэш статики и ISR-страниц
  • Apollo Client — кэш на мобильном браузере

Инвалидация: при изменении товара в Strapi -> webhook -> purge Redis-ключа.

28Оптимистическое обновление корзины — как?

Ответ: Корзина в Redis. При добавлении товара: 1) Оптимистично обновляем UI (Zustand), 2) Отправляем запрос, 3) При ошибке — откатываем UI, 4) При успехе — подтверждаем.

29Нативное приложение или сайт (PWA) для мобильных?

Ответ: Сайт responsive + PWA. Причина: MLM = вирусный рост = минимальное трение. Ссылка → сайт → регистрация = 30 сек. Ссылка → App Store → скачать → регистрация = 2-5 мин, потеря 40-60% конверсии.

PWA функционал: иконка на экране, push-уведомления, офлайн-режим (Service Worker). Настраивается через next-pwa за 2-3 дня.

Когда нативное: при 10K+ активных пользователей, если нужны AR-примерка, сложный офлайн или push через Safari (ограничен в PWA).

7. Масштабирование (Q29–Q30)

29Какие сервисы масштабируем первыми?

Ответ:

  • Product Service — больше всего чтения. Масштабируем горизонтально + Redis.
  • Order Service — нагрузка на запись при оформлении. Масштабируем + rate limiter.
  • Partner Service — пересчёт рангов фоновый, не в горячем пути.
30Docker Compose -> Kubernetes — когда и как?

Ответ: При 10 000+ пользователей и когда运维-нагрузка Docker Compose становится проблемой. Плавно: Docker Compose -> Docker Swarm -> Kubernetes.

Альтернатива: Не переходить на K8s вообще. Использовать управляемые сервисы (Railway, Render), которые масштабируют сами. Экономит сотни часов运维.

8. Развёртывание (Q31–Q34)

31Как выглядит CI/CD для monorepo?

Ответ: GitHub Actions: push в main -> запуск тестов для всех сервисов. Если только сервис X изменён -> деплоим только X (path filter). Frontend -> автодеплой Vercel. Backend -> build Docker -> push GHCR -> deploy Railway.

32Blue-green или canary деплой?

Ответ: На MVP — rolling update (дефолт Railway/Vercel). Canary на фазе 2–3: 10% трафика на новую версию -> мониторинг -> 100%.

33Где храним секреты (API-ключи, пароли БД)?

Ответ: GitHub Secrets для CI/CD. Railway/Vercel environment variables для продакшена. .env.local для локальной разработки (в .gitignore).

Критично: Автоматический сканер секретов в CI (gitleaks) — блокируем коммиты с accidental secrets.
34Как работаем с разными средами (dev, staging, prod)?

Ответ:

  • Local: Docker Compose с пустой PostgreSQL -> drizzle-kit generate -> drizzle-kit migrate -> seed script
  • Dev: Shared dev-окружение на Railway
  • Staging: Копия prod, но с тестовыми данными
  • Prod: Боевая среда

10. Тестирование (Q37–Q39)

37Какой набор тестов нужен?
  • Unit (Jest) — 70%+ покрытия бизнес-логики (расчёт бонусов, рангов)
  • Интеграционные (Supertest) — реальная БД, не моки. Проверяем эндпоинты + Drizzle.
  • E2E (Playwright) — 3–5 критичных сценариев: регистрация -> покупка -> бонусы -> вывод
38Как тестируем логику партнёрских бонусов?

Ответ: Это самая критичная логика. Тестов больше, чем production-кода:

  • Unit-тесты для каждого расчёта бонусов (каждый ранг, каждый уровень)
  • Интеграционные: заказ -> начисление -> hold -> подтверждение
  • Property-based тесты (fast-check): сумма бонусов <= сумма заказа
Критично: Ошибка в расчёте бонусов = финансовые потери. Тесты здесь не опциональны.
39Как мокаем внешние сервисы?
  • ЮKassa — MSW (Mock Service Worker) или nock для мока HTTP
  • СДЭК — мок с фиктивными статусами доставки
  • Email/SMS — не отправляем реально, пишем в лог

11. Мониторинг и логирование (Q40–Q42)

40Как мониторим работоспособность?
  • Sentry — отлов ошибок фронтенда и бэкенда. Алерты в Telegram.
  • Health check — GET /health на каждом сервисе. Railway мониторит автоматически.
  • UptimeRobot — бесплатно, 50 мониторов, проверка каждые 5 мин.
41Куда пишем логи?

Ответ: JSON-логирование, структурированные логи (winston). Каждый запрос имеет correlationId — можно отследить путь через все сервисы. На MVP — стандартный вывод Docker (Railway показывает). На фазе 2 — ELK-стек или Loki.

42Какие бизнес-метрики мониторим?
  • Конверсия визит -> заказ
  • Средний чек
  • Количество регистраций по промокодам в день
  • Сумма начисленных бонусов за день
  • Сумма выплаченных бонусов за неделю
  • Количество возвратов и откатов бонусов

12. Команда и процессы (Q43–Q46)

43Monorepo или polyrepo?

Ответ: Monorepo (pnpm + turborepo). Преимущества:

  • Один PR = изменение во всех сервисах
  • Shared Types — общая библиотека типов
  • Единый CI
Полирепо: когда команды независимые и деплоят не синхронно. Для нас на старте — monorepo проще. При росте до 5+ команд — рассмотрим.
44Kanban или Scrum?

Ответ: Kanban. Инструмент: GitHub Projects (бесплатно, привязка к issue/PR). Колонки: Backlog -> In Progress -> Review -> Done. Без спринтов для MVP.

45Чеклист code review?
  • Логика соответствует ТЗ из бизнес-модели?
  • Тесты написаны и проходят?
  • Нет хардкода секретов?
  • Error handling покрывает edge-cases?
  • Миграции БД additive?
  • Логирование достаточно для отладки?
46Какую документацию ведём?
  • Архитектурные решения (ADR) — почему приняли каждое решение
  • API Docs — GraphQL Playground в dev
  • Runbooks — что делать, если сервис X упал
  • README — каждый сервис: как запустить, как тестировать

13. Бизнес-риски (Q47–Q50)

47Юридический риск MLM-модели?

Ответ: MLM легален в РФ при наличии реального товара. Но нужно: оферта (юрист), договоры с самозанятыми, прозрачная механика. Без юриста — высокий риск.

Блокирующий риск: Без юридической консультации не запускать. Это может закрыть весь проект.
48Vendor lock-in от платёжной системы или хостинга?

Ответ: Платёжная система — абстрактный интерфейс (IPaymentGateway). Сменить ЮKassa на другую = реализовать тот же интерфейс. Хостинг — Docker, миграция на другой хостинг = docker-compose up.

49Что если данные партнёрской сети утекут?

Ответ: Минимизация: не храним больше чем нужно (телефон/email — можно удалить). Шифрование PII в БД (bcrypt для чувствительных полей). Ролевой доступ: аналитик не видит телефоны, только агрегаты.

50Что если сеть вырастет в 100 раз за месяц?

Ответ: Архитектура готова к горизонтальному масштабированию:

  • Product Service -> авто-скейлинг на Railway
  • PostgreSQL -> read-реплики
  • Redis -> кластер
  • GraphQL DataLoader предотвращает N+1 запросы

Боттлнек — скорее всего пересчёт рангов (фоновая задача). На 100x — выносим в отдельный сервис с очередью.
