Техническое проектирование интернет-магазина с партнёрской сетью на headless CMS и микросервисной архитектуре

📅 25 мая 2026 📄 Версия 1.0 🎯 Голова: MVP 💡 Паттерн: Микросервисы (поэтапные)

1. Общее описание архитектуры

Тип архитектуры

Headless CMS + Микросервисы

Почему не монолит: У нас разные категории товаров (косметика + БАДы), сложная партнёрская логика (3 уровня, ранги, бонусы) и личный кабинет. Монолит будет расти и станет неподдерживаемым. С микросервисами каждый компонент развивается отдельно и не ломает остальные.

Почему не Serverless: На старте это дорого (платите за каждый вызов). Когда сеть вырастет — переведём часть сервисов в Serverless.

Основные принципы

Принцип Что означает
Разделение ответственности Каждый сервис делает одну вещь и делает её хорошо
Независимое развёртывание Обновление одного сервиса не требует остановки остальных
GraphQL-first для клиентов Один запрос = все данные для страницы, оптимизация для мобильных
REST между сервисами Внутренняя коммуникация — REST, проще и быстрее
MVP-first Начинаем с минимального набора сервисов, добавляем по мере роста
Простота на старте Не используем Kubernetes, пока не нужно. Docker Compose достаточно.

2. Стек технологий

Каждая технология выбрана по критериям: цена (бесплатно на старте), популярность (легко найти разработчиков), простота (новичок поймёт).

Фронтенд (сайт + админка)

Компонент Технология Почему Альтернативы
Фронтенд-фреймворк Next.js 14 Рендеринг на сервере (SSR) → быстрая загрузка, SEO-дружелюбный, большая экосистема Vue.js + Nuxt, Astro
UI-библиотека React + TailwindCSS Готовые компоненты, лёгкая кастомизация, быстро для новичка Material UI, Chakra UI
Админка Refine (на базе React) Встроенные CRUD, таблицы, фильтры — экономит 70% времени на админку Strapi Admin (встроенный), React Admin
Состояние приложения Zustand Простой, лёгкий, для e-commerce достаточно Redux Toolkit, Jotai

Бэкенд (микросервисы)

Компонент Технология Почему Альтернативы
Язык Node.js + TypeScript Один язык для фронтенда и бэкенда, огромная экосистема, легко найти разработчиков Python (FastAPI), Go
Фреймворк API (клиент) GraphQL (Apollo Server) Один запрос = все данные для страницы, нет overfetching/underfetching, критично для мобильных REST (для вебхуков внешних сервисов)
Клиент GraphQL Apollo Client Автоматическое кэширование на клиенте, оптимистичные обновления, работа при слабом интернете urql, SWR + fetch
Межсервисная коммуникация REST Между микросервисами REST проще и быстрее; GraphQL только «фасад» для клиентов gRPC, tRPC

4. Подробное описание микросервисов

Каждый микросервис — это отдельное приложение с своим API, своей базой данных (или схемой в общей БД) и своей областью ответственности.

🛒 Сервис товаров (Product Service)

Что делает: Каталог, категории, цены, наличие, фото

API: GET /products, GET /products/:id, GET /categories

Данные: Таблицы products, categories, product_prices, inventory

Зависимости: Strapi (для контента/описаний), PostgreSQL

Почему отдельно: Каталог — самая читаемая часть, кэшируется в Redis

🛍️ Сервис заказов (Order Service)

Что делает: Корзина, оформление заказа, статусы, история

API: POST /cart, POST /checkout, GET /orders/:id

Данные: Таблицы carts, orders, order_items, payments

Зависимости: Redis (корзина), PostgreSQL (заказы), ЮKassa (оплата)

Почему отдельно: Самая нагруженная часть при покупке, нужно масштабировать

🎫 Сервис партнёрки (Partner Service)

Стек: Python + FastAPI

Что делает: Пользователи, промокоды, бонусы, ранги, цепочки

API: POST /register, POST /apply-promo, GET /bonuses, POST /recalculate-ranks

Данные: users, promo_codes, network_links, bonus_transactions, ranks, rank_history

Зависимости: PostgreSQL, Redis (сессии), Email/SMS (уведомления)

Почему отдельно и Python: Самая сложная логика — графовые вычисления, обходы сети. Python лучше для алгоритмов на графах и статистики.

💰 Сервис выплат (Payout Service) — Фаза 2

Что делает: Расчёт выплат, реестры, интеграция с банком

API: POST /calculate, POST /payout, GET /payouts

Данные: payouts, payout_schedules, bank_accounts

Зависимости: Сервис партнёрки (бонусы), банк (СБП/ЮKassa)

📧 Сервис уведомлений (Notification Service) — Фаза 2

Что делает: Email, SMS, push-уведомления

API: POST /send-email, POST /send-sms, POST /send-push

Зависимости: SMTP-сервер, SMS-шлюз, Firebase (push)

📊 Сервис аналитики (Analytics Service) — Фаза 2

Стек: Python + FastAPI

Что делает: Статистика, воронки, топ-агенты, отчёты

API: GET /stats, GET /leaderboard, GET /funnel

Зависимости: Все сервисы (читает данные), pandas для анализа

Разделение баз данных

Каждый сервис имеет свою схему в PostgreSQL (или свою БД на масштабе):

Сервис Схема БД Таблицы
🛒 Товары product_service products, categories, product_prices, inventory, media
🛍️ Заказы order_service carts, orders, order_items, payments, shipping
🎫 Партнёрка partner_service users, promo_codes, network_links, bonus_transactions, ranks
💰 Выплаты payout_service payouts, payout_schedules, bank_accounts

6. API контракты (основные эндпоинты)

Клиентский API — GraphQL. Все запросы с фронтенда идут через /api/graphql — один запрос получает всё, что нужно для страницы. Между микросервисами используется REST (внутренняя коммуникация). Вебхуки от ЮKassa/СДЭК — REST.

Пример: страница товара одним GraphQL-запросом

query ProductPage($id: ID!) {
  product(id: $id) {
    name
    price
    description
    images { url alt }
    category { name slug }
    reviews(limit: 5) {
      author
      rating
      text
      date
    }
  }

    name price image
  }

    cartCount
    bonuses
    rank
  }
}

/* Результат: 1 запрос ~120ms вместо 4 REST-запросов ~350ms */
/* Экономия: 65% времени загрузки на мобильном интернете */

7. Ключевые потоки данных

🔗 Поток: Регистрация по промокоду

Что происходит при регистрации нового пользователя
Пользователь (мобильный браузер)
→ mutation Register($input: RegisterInput!) { register(input: $input) { user { id rank } ownPromoCode } }
API Gateway (GraphQL Resolver)

1. Вызывает Partner Service (POST /internal/auth/register)

2. Проверяет промокод → находит владельца (агент X)

3. Создаёт пользователя → уровень 2 (если нашёл агента) или 3

4. Записывает network_links: upline=agent X, downline=новый пользователь

5. Генерирует свой промокод → записывает в promo_codes

PostgreSQL (users, network_links, promo_codes)
← Ответ GraphQL: { success: true, user: {...}, ownPromoCode: "USER-XYZ" }

💰 Поток: Покупка → начисление бонусов

Что происходит при оформлении заказа
Покупатель (мобильный браузер)
→ mutation Checkout($input: CheckoutInput!) { checkout(input: $input) { order { id total } bonuses { amount user } } }
API Gateway (GraphQL Resolver)

1. Вызывает Order Service → создаёт заказ

2. Вызывает Product Service → проверяет наличие

3. Вызывает Partner Service → находит цепочку, начисляет бонусы

Partner Service

4. Находит цепочку: покупатель → реализатор → агент

5. Читает ранги → считает бонусы с множителями

6. Создаёт bonus_transactions (status: pending, hold_until: +7 дней)

PostgreSQL (orders, bonus_transactions)
← Ответ GraphQL: { success: true, order: {...}, bonusesCalculated: [...] }

⚡ Поток: Страница товара (оптимизация для мобильных)

Один GraphQL-запрос вместо 4 REST-запросов
Мобильный браузер
→ query { productPage(id: "123") { ... } }
API Gateway

Параллельно (Promise.all):

→ Product Service: товар + категория

→ Product Service: похожие товары

→ Partner Service: моя корзина + бонусы

PostgreSQL + Redis (кэш товаров)
← 1 ответ ~120ms (вместо 4× запросов ~350ms с REST)

🏆 Поток: Ежемесячный пересчёт рангов

1-го числа каждого месяца (крон-задача)
GitHub Actions (крон)
→ POST /api/ranks/recalculate
Partner Service

1. Для каждого пользователя считает: продажи за 3 месяца

2. Сравнивает с условиями рангов

3. Если ранг изменится → обновляет rank и rank_multiplier

4. Записывает rank_history (для аналитики)

5. Отправляет уведомления (сервис уведомлений)

PostgreSQL (users, ranks, rank_history)
