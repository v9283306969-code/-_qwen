## Когда применять

После завершения крупного этапа или при возобновлении работы после перерыва. Используется для **систематической проверки** что фактическое состояние проекта соответствует документации (SPEC, ROADMAP, PROJECT-LIFECYCLE) и выявления скрытых проблем.

## Процедура

### Шаг 1: Рекурсивный скан файлов

Собрать полный список файлов проекта (исключая `.git`, `node_modules`, `.qwen`, `pnpm-store`). Для каждого файла отметить:
- ✅ Присутствует и соответствует назначению
- ⚠️ Присутствует но пустая директория (содержимое не создано — это нормально для будущих этапов)
- ❌ Присутствует но лишний/артефакт (не ожидаемый документацией)
- 🔴 Ожидается по SPEC но отсутствует (будущие шаги)

### Шаг 2: Сравнение с документацией

Для каждого SPEC-файла (SPEC-01..XX):
1. Прочитать раздел «Файлы проекта» (таблица ожидаемых файлов)
2. Проверить каждый файл физически на диске
3. Отметить в таблице ✅ если создан, `[ ]` если ещё нет

### Шаг 3: Проверка .gitignore на «потерянные» файлы

**Критическая проверка:**
```bash
# Для каждого .yml/.env/.toml файла в проекте:
git ls-files --error-unmatch <filename> 2>&1
# Если "did not match" — файл в .gitignore и НЕ попадёт в репозиторий
```

Это обнаруживает файлы, которые физически существуют, но `not tracked by git` — самый распространённый источник скрытых проблем.

### Шаг 4: Проверка консистентности конфигов

| Что проверить | Как | Критерий |
|---|---|---|
| Версии compose-файлов | Сравнить `version:` в docker-compose.yml vs .prod.yml vs .dev.yml | Должны быть одинаковы (или все без version) |
| Lock-файлы | `ls *lock*` | Для pnpm — только `pnpm-lock.yaml`. `package-lock.json` — лишнее |
| Deploy vs Docker Compose | Есть ли секции `deploy:` в docker-compose? | `deploy` работает только в Docker Swarm, не в `docker compose up` |
| `.eslintrc*` в COPY | `ls .eslintrc*` | Если файлов нет — glob в Dockerfile вызовет ошибку сборки |
| pnpm `--frozen-lockfile` | Есть ли `pnpm-lock.yaml`? | Если нет — `--frozen-lockfile` упадёт |

### Шаг 5: Проверка CI pipeline

1. Прочитать `.github/workflows/ci.yml`
2. Выявить все `run:` команды — что они фактически делают?
3. Проверить нет ли `|| true` / `continue-on-error: true` которые маскируют проблемы
4. Сравнить стадии CI с диаграммой из SPEC (например, deploy в SPEC-05 но закомментирован в ci.yml)
5. Проверить что ветки в `on.push.branches` соответствуют branching strategy

### Шаг 6: Классификация проблем

| Приоритет | Что | Действие |
|---|---|---|
| 🔴 Критические | Файлы, не попадающие в git; несовместимые версии; broken configs | **Исправить сейчас** |
| ⚠️ Документационные | SPEC диаграмма ≠ реальность; незадокументированные решения | **Задокументировать в SPEC и memory** |
| ℹ️ Расширения | Сделано больше чем SPEC требовал (полезные дополнения) | **Записать в SPEC как новое** |

### Шаг 7: Фиксация критических проблем

Для каждой 🔴 проблемы:
1. Исправить файл
2. Добавить комментарий `# ⚠️` в код если это workaround
3. Задокументировать в разделе «Исправлено после аудита» в SPEC

### Шаг 8: Создание audit-файла

Создать `docs/audit-YYYY-MM-DD.md` с:
- Результаты всех проверок
- Список пустых директорий (и почему это НЕ ошибка)
- Сводка: сколько исправлено, сколько задокументировано
- Общее состояние проекта

### Шаг 9: Обновление PROJECT-LIFECYCLE.md и memory

Обновить `current_work.md` (memory) с:
- Полный статус всех этапов (0-N)
- Все ⚠️ отклонения scaffold от SPEC
- Все 🔧 исправления аудита
- 10+ точек восстановления контекста (ROADMAP, SPEC, CI, и т.д.)

## Типичные критические проблемы (из практики)

| # | Проблема | Влияние | Фикс |
|---|---|---|---|
| 1 | Production конфигурация в `.gitignore` | Файл не попадёт в репозиторий при clone | Удалить из .gitignore, `git add` |
| 2 | `version: '3.9'` в одном compose-файле, `version` нет в другом | Несоответствие, warning в CI | Удалить version из всех файлов |
| 3 | `deploy:` секции в docker-compose.yml | Работает только в Swarm, создаёт ложное ожидание | Добавить warning comment |
| 4 | Лишний lock-файл (`package-lock.json` при pnpm) | Создаёт путаницу, может конфликтовать | Удалить |
| 5 | `.eslintrc*` в `COPY` Dockerfile но файлов нет | Сборка падает | Убрать glob из COPY |
| 6 | `--frozen-lockfile` без `pnpm-lock.yaml` | Сборка падает | `--no-frozen-lockfile` пока нет lock-файла |
